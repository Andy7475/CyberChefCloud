# How to Add a New Google Cloud Ingredient

When adding a new Google Cloud integration (ingredient or operation) to CyberChef Cloud, it's important to strictly separate core authentication logic from specific application logic.

## 1. Code Structure & Placement

- **Core Cloud Components (`src/core/lib/GoogleCloud.mjs`)**: This file should *only* contain common Google Cloud logic that is shared across multiple ingredients, namely credential management (`get_gcp_credentials`, `set_gcp_credentials`) and applying authentication headers to `fetch` requests (`applyGCPAuth`). Do not place specific API feature implementations in this core library.
- **Specific Ingredient Code (`src/core/operations/[OperationName].mjs`)**: The majority of the logic for any new Google Cloud ingredient—such as making the specific API calls, handling responses, parsing data, and long-running operation polling—must be encapsulated inside its separate `*.mjs` file located in the `src/core/operations/` directory. Be sure to import `applyGCPAuth` when you need to attach credentials to outbound requests.

## 2. Authorisation Flows

When dealing with Google Cloud APIs from the browser, any REST request originates directly from the web worker via `fetch()`. Keep these architectural points in mind:

- **Browser-Side Secret Exposure**: Secrets are visible to the browser. API Keys must have HTTP Referrer restrictions. Temporary OAuth tokens are preferred as they expire and reduce the risk of credential leakage.
- **Application Default Credentials (ADC) Quota Project**: When using a short-lived OAuth token generated by an End User account, calling APIs may throw a 403 error requiring a "quota project." The Quota Project is configured **once** in the `Authenticate Google Cloud` ingredient and is stored alongside the credentials. `applyGCPAuth` in `GoogleCloud.mjs` automatically attaches it via the `x-goog-user-project` HTTP header on every request. **Downstream ingredients must NOT expose a Quota Project argument** — doing so would be redundant and breaks the single-point-of-configuration principle.
- **UI Argument Masking (The `toggleString` Pitfall)**: To protect secrets from being displayed in plain text, use the `toggleString` data type for authentication arguments. **Important:** Remember that a `toggleString` returns an object (`{ "string": "...", "option": "..." }`), not a raw string. The `fetch` calls must safely parse these masked keys. Avoid silently crashing by ensuring you handle `authStringObj.string`.

## 3. Content Security Policy (CSP) & Whitelisting 

CyberChef Cloud employs a strict Content Security Policy defined in `src/web/html/index.html`. 

If your new ingredient communicates with a Google Cloud API endpoint that hasn't been used before (e.g., `https://vision.googleapis.com`), you **must** whitelist this endpoint:
1. Open `src/web/html/index.html`.
2. Locate the `<meta http-equiv="Content-Security-Policy">` tag.
3. Append your new endpoint to the `connect-src` directive.
4. Add the endpoint to the `docs/AuthorizedEndpoints.md` file to keep the documentation current.

## 4. Input & Output Strategies (Google Cloud Storage)

When a Google Cloud ingredient interacts with large media or datasets that cannot fit in browser memory, it must accept Google Cloud Storage (GCS) URIs as input and write back to GCS as output.

To ensure CyberChef remains an effective pipelining/orchestration tool (e.g., passing thousands of files through a `Fork` operation), all GCS-writing ingredients must adhere to the **Hybrid Target Directory Pattern**:

### Required Arguments:
1. **Output Mode (Dropdown)**: `["Return to CyberChef", "Write to GCS"]`
   - *Return to CyberChef*: Used only when returning raw JSON or parsed text (e.g., "Cat, Dog") directly to the browser output pane.
   - *Write to GCS*: Used when mutating media or producing large output files.
   
2. **Output Directory (String)**: `gs://my-bucket/outputs/`
   - **If blank (Default Behaviour)**: Write the output file to the *exact same directory* as the input file, but you **MUST** suffix the filename (before the extension) with a concise, hardcoded operation string (e.g., `_ccc_ocr`, `_ccc_redact`, `_ccc_stt`).

**Note:** You **MUST** use the `generateGCSDestinationUri(inputUri, destDir, suffix, extensionOverride)` utility function exported from `src/core/lib/GoogleCloud.mjs` to systematically parse the URIs and generate the correct final destination according to these rules.
   
## 5. Returning GCS URIs to Output

When "Write to GCS" is executed successfully, the ingredient **must** return the full destination GCS URIs (`gs://...`) back to the CyberChef browser output. This gives the user a handy list of the newly generated files, ready to be piped directly into the next ingredient in their workflow.

**Example of Default Suffix Behaviour:**
Input: `gs://source-bucket/images/scanned_doc.pdf`
Blank Output Directory → Writes to: `gs://source-bucket/images/scanned_doc_ccc_ocr.pdf`
CyberChef Output Box displays: `gs://source-bucket/images/scanned_doc_ccc_ocr.pdf`
